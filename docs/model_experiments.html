The boosting trio clearly dominates the bake-off. On the validation split, LightGBM edges out as best by RMSE with 5.639 (MAE 2.743, R² 0.9182), narrowly ahead of XGBoost at 5.643 RMSE (MAE 2.721, R² 0.9181) and CatBoost at 5.715 RMSE (MAE 2.747, R² 0.9160). The margins between LGBM and XGB are within noise-level for a single split, so both are competitive winners; CatBoost is close but consistently a touch behind in this configuration.

Linear baselines underfit relative to boosters. LinearRegression and Ridge(α=1.0) both land at 6.970 RMSE and 3.661 MAE (R² 0.8751). That’s a sizable gap from the boosters (~+1.3 RMSE), indicating substantial non-linear structure and feature interactions in the data that linear models can’t capture even with regularization.

Runtime/throughput trade-offs favor LightGBM and XGBoost over CatBoost here. Fit time: LGBM ~28.2s, XGB ~34.5s, CatBoost ~117.9s. Prediction latency was low for all boosters but fastest for XGB (~0.58s) and LGBM (~1.39s) vs CatBoost (~3.87s). Given near-identical accuracy between LGBM and XGB, their shorter fit+predict times make them strong choices for iteration and deployment.

The mean baseline (RMSE 19.782, MAE 13.611, R² ≈ 0) confirms that learned models extract substantial signal.

def fit_time_predict(name, est, Xtr, ytr, Xva, yva, is_baseline: bool = False):
    if is_baseline:
        t0 = time.time()
        yhat = np.full_like(yva, ytr.mean(), dtype=float)
        return dict(
            Model=name,
            MAE=mean_absolute_error(yva, yhat),
            RMSE=root_mean_squared_error(yva, yhat),
            R2=r2_score(yva, yhat),
            Fit_s=0.0,
            Pred_s=time.time() - t0,
        )

    t0 = time.time()
    est.fit(Xtr, ytr)
    fit_s = time.time() - t0

    t1 = time.time()
    yhat = est.predict(Xva)
    pred_s = time.time() - t1

        Model=name,

        RMSE=rmse(yva, yhat),
        R2=r2_score(yva, yhat),

        Pred_s=pred_s,
    )

In this step I used Optuna to tune LightGBM. The search ran for 50 trials with a TPE sampler, testing different values for key parameters like num_leaves, max_depth, min_child_samples, subsample, colsample_bytree, regularization terms, learning_rate, and n_estimators. The goal was to minimize RMSE on the validation set, with early stopping (200 rounds) to avoid overfitting and wasted training time. A fixed random seed ensured that the search results are reproducible.

The parameter ranges were chosen to balance model flexibility (e.g. larger trees with num_leaves and max_depth) against regularization (e.g. reg_alpha, reg_lambda, min_child_samples). Using subsample and colsample helps reduce variance and training cost. Since all trials use the same validation split, results are directly comparable, but to avoid overfitting to one period it will be useful to test the best parameters on another month or with time-aware CV.

N_TRIALS = 50
EARLY_STOPPING_ROUNDS = 200
TIMEOUT = None
EXP_DIR = BASE_ARTIFACTS_DIR / "model_experiments" / "optuna_lgbm"
EXP_DIR.mkdir(parents=True, exist_ok=True)
STUDY_NAME = "lgbm_regression_optimization"

np.random.seed(SEED)
random.seed(SEED)

# Study
sampler = optuna.samplers.TPESampler(seed=SEED)

study = optuna.create_study(
    direction="minimize",
    study_name=STUDY_NAME,
    sampler=sampler,
)


# Objective
def objective(trial: optuna.trial.Trial) -> float:
    params = {
        "n_estimators": trial.suggest_int("n_estimators", 500, 3000, step=250),
        "learning_rate": trial.suggest_float("learning_rate", 0.005, 0.2, log=True),
        "num_leaves": trial.suggest_int("num_leaves", 31, 511, step=32),
        "max_depth": trial.suggest_int("max_depth", 4, 16),
        "min_child_samples": trial.suggest_int("min_child_samples", 10, 200),
        "subsample": trial.suggest_float("subsample", 0.6, 1.0),
        "colsample_bytree": trial.suggest_float("colsample_bytree", 0.6, 1.0),
        "reg_alpha": trial.suggest_float("reg_alpha", 1e-8, 10.0, log=True),
        "reg_lambda": trial.suggest_float("reg_lambda", 1e-8, 10.0, log=True),
        "min_split_gain": trial.suggest_float("min_split_gain", 0.0, 1.0),
        "random_state": SEED,
        "n_jobs": 1,
        "feature_fraction_seed": SEED,
        "bagging_seed": SEED,
        "data_random_seed": SEED,
        "verbosity": -1,
    }

    model = lgb.LGBMRegressor(**params)

    callbacks = [lgb.early_stopping(EARLY_STOPPING_ROUNDS, verbose=False)]

    model.fit(
        X_train_enc,
        y_train,
        eval_set=[(X_val_enc, y_val)],
        eval_metric="rmse",
        callbacks=callbacks,
    )

    preds = model.predict(X_val_enc)
    score = root_mean_squared_error(y_val, preds)
    trial.set_user_attr("valid_rmse", score)

    del model
    gc.collect()

    return score


# Optimize
study.optimize(
    objective,
    n_trials=N_TRIALS,
    timeout=TIMEOUT,
    n_jobs=1,
    show_progress_bar=True,
    gc_after_trial=True,
)

print("Best RMSE:", study.best_value)
print("Best params:", study.best_params)

# Save: best params (JSON)
(EXP_DIR / "best_params.json").write_text(json.dumps(study.best_params, indent=2))

def trial_to_dict(t: optuna.trial.FrozenTrial):
    return {
        "number": t.number,
        "value": t.value,
        "state": str(t.state),
        "params": t.params,
        "user_attrs": t.user_attrs,
        "duration": t.duration.total_seconds() if t.duration else None,
        "datetime_start": t.datetime_start.isoformat() if t.datetime_start else None,
        "datetime_complete": (
            t.datetime_complete.isoformat() if t.datetime_complete else None
        ),
    }

This cell compares training on TRAIN only vs TRAIN+VAL and evaluates both on the same TEST set. Merging VAL into training clearly helps: RMSE drops from 6.1591 → 5.6032 (~9% better), MAE from 3.0245 → 2.9456 (~2.6% better), and R² rises from 0.9044 → 0.9209 (+0.0165). These gains suggest the model benefits from the extra examples in VAL without overfitting.

Takeaway: once hyperparameters are chosen, it’s a viable option to retrain on TRAIN+VAL to give the model more signal before the final TEST check. The TRAIN+VAL model is the one to keep for deployment (model.joblib), and the saved metrics JSONs allow quick auditing.
